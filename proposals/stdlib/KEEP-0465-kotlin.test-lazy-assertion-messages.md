# Lazy error messages for kotlin.test assertions

* **Type**: Standard Library API proposal
* **Author**: Filipp Zhinkin
* **Contributors**: Ilya Gorbunov, Brian Norman
* **Status**: Experimental in 2.4.20
* **Related issues**: [KT-31305](https://youtrack.jetbrains.com/issue/KT-31305) 
* **Discussion**: TBD

## Summary

Extend existing `kotlin.test` assertion functions API with overloads accepting a lambda generating an error message.

## Motivation and use cases

Assertions are an indispensable part of almost every unit or functional test.
And even though every test is filled with them, they are rarely (ideally, never) fired.
However, formatting a detailed failure description may come with the price, and currently that price has to be paid
for each and every test execution, disregard the test result.

At the same time, it is clear that in tests you can find across existing code bases (including Kotlin's Standard Library)
error messages are rarely passed to assertions, and even if they do, it's likely some constant string literals:

```kotlin
assertTrue(file.exists())
// ...
assertFalse(testDirectory.exists(), "Directory was not deleted")
```
[^from the stdlib's RecursiveDeletionTest.kt](https://github.com/JetBrains/kotlin/blob/ffca2b40187d0f427148fcc5312d7a92b9dbc1b0/libraries/stdlib/jdk7/recursiveDeletionTest/RecursiveDeletionTest.kt#L40)

But error messages could be a bit more involved and require more time to construct:
```kotlin
val actual = xml.decodeFromString<NodeTag>(expectedXml)
assertEquals(2, actual.content.size, "Unexpected content: ${actual.content.joinToString()}")
```
[^from xmlutil's regression test suite](https://github.com/pdvrieze/xmlutil/blob/c02bf917fcc716f86a23bf73eb9734e5d61ae003/serialization/src/commonTest/kotlin/nl/adaptivity/xml/serialization/regressions/EntityInNode291.kt#L94)

```kotlin
assertTrue("Failed for $places and elements ${number.contentToString()}") {
    // ...
}
```
[^from kotlin-multiplatform-bignum test suite](https://github.com/ionspin/kotlin-multiplatform-bignum/blob/e9544397e955eec082eb3dc2720584cb6dd92a37/bignum/src/jvmTest/kotlin/com/ionspin/kotlin/bignum/integer/base32/BigInteger32JavaBitwiseTest.kt#L64)

And even if users don't write such elaborative error explanations on their own, 
[the Kotlin Power-Assert compiler plugin](https://kotlinlang.org/docs/power-assert.html) implicitly generates
rich detailed error explanations when applied to a project. 

For both manually formatted messages and explanations generated by the Power-Assert,
it may take some measurable amount of time to create a failure description at runtime,
and, ideally, that time should be spent only when an assertion is about to fail.  

Akin to logging, some assertion libraries and test frameworks support assertion functions accepting 
a message generator that will be called only to format an error message when the checked condition does not hold
and the assertion is considered failed:
```java
// Lazily evaluates generateFailureMessage('a','b').
assertTrue('a' < 'b', () -> generateFailureMessage('a','b'));
// ...
private static String generateFailureMessage(char a, char b) {
    return "Assertion messages can be lazily evaluated -- "
            + "to avoid constructing complex messages unnecessarily." + (a < b);
}
```
[^ from JUnit's docs](https://docs.junit.org/6.1.0/writing-tests/assertions.html)


Right now, `kotlin.test` provides assertion functions (like `assertEquals`) accepting only a pre-formatted error message 
(or no message at all).

To improve execution times for tests formatting complex error explanations 
(either manually, or by means of the Power-Assert plugin), 
`kotlin.test` assertions API should be extended by adding overloads accepting a lambda function 
generating an error message on an assertion failure.

## Similar API review

There are several libraries providing assertions that could be utilized by Kotlin projects.

Just to name a few:
- [JUnit](https://junit.org)
- [TestNG](https://testng.org)
- [AssertJ](https://assertj.github.io/doc/)
- [Truth](https://truth.dev)
- [Kotest](https://kotest.io)
- [Atrium](https://github.com/robstoll/atrium)
- [Spek](https://www.spekframework.org)

These libraries use different approaches when it comes to asserting conditions. 
Some of them provide API similar to `kotlin.test` (`JUnit`, `TestNG`).
Others use fluent or matcher-based API (`Kotest`, `Truth`, etc).

Among `kotlin.test`-alike libraries, only `JUnit` supports lazy messages starting from its 5th version.

And for fluent and matcher-based APIs, the necessity to provide an explicit failure description
is less critical, as these APIs could either build all the context using built-in assertion builders,
or allow developers to implement their own matchers.

## Proposed API

Existing assertion functions could be split into several groups.

### Unconditional assertions

This group consists of the [fail](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/fail.html) 
function, which, as the name suggests, fails the test.
As this function always fails, it makes no sense to provide an overload accepting a message generator: 
it will be invoked anyway.

### Functions evaluating a lambda block

There are several functions that either compare the result of some lambda with an expected value 
(like [expect](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/expect.html),
or one of the [assertFalse](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-false.html) overloads), 
or evaluate a block of code and assert certain behavior (like [assertFails](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-fails.html)).

While it makes sense to generate an error message only when these assertion functions fail,
it is not obvious how to introduce yet another lambda argument and keep the API ergonomic.

For now, it was decided to abstain from providing additional overloads for these functions.

### Assertion functions accepting expected and actual values

A vast majority of existing assertion functions belong to this group:
- [assertTrue](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-true.html), [assertFalse](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-false.html)
- [assertEquals](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-equals.html), [assertNotEquals](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-not-equals.html)
- [assertSame](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-same.html), [assertNotSame](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-not-same.html)
- [assertIs](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-is.html), [assertIsNot](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-is-not.html)
- [assertContains](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-contains.html)
- [assertContentEquals](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-content-equals.html)
- [assertNull](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-null.html), [assertNotNull](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/assert-not-null.html)

All these functions (except `assertNotNull`) will get an overload accepting a lambda message. For example:
```kotlin
// Existing function:
public fun assertTrue(actual: Boolean, message: String? = null)
// New overload:
public fun assertTrue(actual: Boolean, lazyMessage: () -> String)
```

New functions will be placed in the same `kotlin.test` package along with the existing assertion functions and
should be annotated as `@ExperimentalKotlinTestApi` until their stabilization.

### Why is there no overload for `assertNotNull` 

There is an existing `assertNotNull` overload accepting a lambda block that 
is called when the asserted value is not null. Adding yet another overload accepting a lambda as a trailing argument
will cause overload resolution ambiguity in certain cases, so we can not proceed with it.

At the same time, the existing overload proved to be misused (see [KT-86760](https://youtrack.jetbrains.com/issue/KT-86760))
and there's a plan to deprecate and remove it. After that, an overload accepting a message generator could be introduced.

### `Asserter` evolution

Under the hood, assertion functions end up using an instance of 
the [kotlin.test.Asserter](https://kotlinlang.org/api/core/kotlin-test/kotlin.test/-asserter/) interface.
While the interface provides facilities to implement complex assertion functions, its implementations are aimed
to provide integration with platform-specific test frameworks.

It makes sense to extend the interface with functions accepting a lazy message generator (especially, given that
the interface already provides a few such functions). However, extending the interface with experimental functions,
or adding an intermediate experimental sub-interface may cause source and binary compatibility issues for 
both `Asserter` users and implementors if we decide to change the experimental API.

At the same time, only `JUnit5Asserter` natively supports lazy message generators, and the lazy messages could
be supported even without such native support.

As a result, the decision was made to abstain from changing the `Asserter` interface for now.

Once assert*Something* overloads accepting a message generator are stable, 
the `Asserter` interface can be updated to provide similar overloads for all its functions.
These overloads can have a default implementation, thus existing implementation will not be affected.

## Reference implementation

The new functions were implemented and integrated into Kotlin `2.4.20` in [#6242](https://github.com/JetBrains/kotlin/pull/6242).

## Roadmap

* [x] Implement new assertion functions
* [ ] Collect users feedback
* [ ] Based on feedback, consider adding overloads for functions that did not receive new overloads
* [ ] Start deprecating the existing `assertNotNull` overload with a lambda parameter in Kotlin `2.5.0`
* [ ] Introduce `assertNotNull` overload with a lazy message generator
* [ ] Stabilize the API
* [ ] Add overloads accepting a message generator to the `Asserter` interface
