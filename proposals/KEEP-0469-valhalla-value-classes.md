# Project Valhalla value classes support

* **Type**: Design proposal
* **Author**: Evgeniy Zhelenskiy
* **Status**: Submitted
* **Discussion and feedback**: TODO
* **Related YouTrack issues**: [KT-84904](https://youtrack.jetbrains.com/issue/KT-84904), [KT-87569](https://youtrack.jetbrains.com/issue/KT-87569)

# Abstract

This KEEP describes experimental support for compiling Kotlin value-like declarations to
[Project Valhalla](https://openjdk.org/projects/valhalla/) value classes on the JVM, as specified by
[JEP 401: Value Classes and Objects](https://openjdk.org/jeps/401).

Kotlin already unboxes inline `value class`es where it can, but only in a limited set of situations, and every
class it compiles — including `value class`es — is still an ordinary JVM identity class at run time. With this
support, a selected set of value-like declarations is instead compiled as Project Valhalla value classes, so
that their instances behave as value objects and can be optimized by the JVM in many more cases than Kotlin can
achieve on its own.

The support is gated behind a new compiler flag, `-Xvalhalla-support`, which selects one of four progressively
wider modes (`none`, `primitives`, `primitivesAndFullValueClasses`, `allValues`), and — for every mode other
than `none` — behind the JVM preview machinery on both the compiler and the runtime. Interoperability with Java
value classes is supported as well.

This is an **experimental** feature intended for trying Project Valhalla out with Kotlin; it is not meant for
production use, and its shape may change as JEP 401 evolves toward integration into a mainline JDK.

# Overview

Project Valhalla value classes and their semantics are specified by [JEP 401](https://openjdk.org/jeps/401);
the value-like declarations Kotlin can express are covered by
[KEEP-0454](https://github.com/Kotlin/KEEP/blob/main/proposals/KEEP-0454-better-immutability-value-classes-MFVC.md).
This proposal only concerns compiling such declarations to JEP 401 value classes on a Valhalla-capable JVM, so
that their instances behave as value objects at run time.

## What this feature does

When enabled, the compiler compiles a selected set of identity-free declarations as JEP 401 value classes
rather than as ordinary identity classes. Concretely, for each declaration that is treated as a Valhalla value
class the compiler:

- emits a class file without the legacy identity marker (the `ACC_SUPER` access flag), which is how the JVM
  recognizes a value class and reports `Class.isValue() == true`;
- marks the class's instance fields *strict* (they must be assigned before the superclass initializer runs);
- records the JEP 401 `LoadableDescriptors` attribute for fields whose type is a value class or a value-based
  class, so the JVM can preload and flatten them.

Which declarations are treated this way is controlled by the mode (see [Modes](#modes)). Declarations that are
*not* treated as value classes keep the ordinary identity representation (`ACC_SUPER` set, non-strict fields),
exactly as Kotlin compiles them today.

The feature also makes Kotlin and Java value classes interoperate in both directions and keeps the
`LoadableDescriptors` attribute correct even when class files are regenerated (for example, when the body of an
inline function is re-emitted at a cross-module call site).

## Experimental status

Project Valhalla value classes are a *preview* feature of the JVM, and Kotlin's support for them is
correspondingly **experimental**:

- It exists only to let people try Project Valhalla out with Kotlin and to co-evolve with JEP 401. It is not
  supported for production, and the set of modes, the flag, and the generated bytecode may all change.
- Every mode other than `none` requires the JVM preview machinery to be explicitly enabled, at both compile
  time and run time (see [Preview requirements](#preview-requirements)).
- Only the default `none` mode works on an ordinary JDK; the other modes require a Valhalla-capable JDK.

## Getting a Valhalla-capable JDK

The other modes require a JDK that implements JEP 401. Such builds are published as **Project Valhalla
Early-Access builds** at [https://jdk.java.net/valhalla/](https://jdk.java.net/valhalla/): download an
early-access JDK, unpack it, and point the compiler and the runtime at it. At the time of writing these builds
implement *JEP 401: Value Classes and Objects (Preview)* on top of a recent JDK feature version, corresponding
to JVM target 27. Because preview class files are tied to the exact JDK feature version that produced them, the
compile-time JVM target must match the Valhalla JDK used to run the result.

Unlike the specification, Kotlin does not pin support to a fixed JDK major version (the specification fixes it
at 67 at the time of writing). Instead, when a Valhalla mode is enabled, the compiler raises the Java language
level to the highest available one rather than to a specific number. Java may increase that version in a future
release, so leaving it unpinned keeps the support future-proof.

# How to use

## The `-Xvalhalla-support` flag

Valhalla support is selected with a single JVM compiler flag:

```
-Xvalhalla-support={none|primitives|primitivesAndFullValueClasses|allValues}
```

The four modes form a chain: each one compiles everything the previous one did as a Valhalla value class, plus
one more category of declarations. The mode must match the capabilities of the target JDK — `none` for an
ordinary JDK, any other mode for a Valhalla-capable one.

## Preview requirements

Because JEP 401 is a JVM preview feature:

- **At compile time**, every mode other than `none` additionally requires Kotlin's JVM preview flag,
  `-Xjvm-enable-preview`. Compiling with `none` requires neither preview nor a Valhalla JDK.
- **At run time**, the produced class files are preview class files, so running them requires the standard
  `--enable-preview` flag on a Valhalla-capable JDK of the matching feature version. Running them on a
  non-preview or non-Valhalla JVM will be rejected by the class loader.

## Modes

### `none` (default)

This is the default. It means the target JDK is *not* Valhalla-compatible: no declaration is compiled as a
Valhalla value class, and the compiler emits regular JVM bytecode. Every class — including `value class`es —
keeps its identity: the identity marker stays set and instance fields are non-strict. This is the
representation Kotlin has always produced, and it requires neither the preview flags nor a Valhalla JDK.

### `primitives`

The most conservative Valhalla mode. Only the built-in primitives are compiled to and behave as Valhalla value
classes; **no** user-declared class changes representation. A plain `value class`, an `@JvmInline value class`,
a `@JvmRecord value class`, and abstract or sealed value classes all keep the ordinary identity representation
in this mode — none of them becomes a value class, and no instance field is strict:

```kotlin
value class Point(val x: Int, val y: Int)      // still an identity class
@JvmInline value class Id(val value: Int)       // still an identity class
```

The only thing that changes for a user is **how primitives behave**. On a Valhalla JDK the primitive wrapper
types (`java.lang.Integer` and friends) are value classes, so a *boxed* primitive no longer has reliable
identity, and code that relies on the identity of boxed primitives behaves differently. For example:

```kotlin
fun check(): String {
    val a = 1000
    val b = 1000
    // '===' is not allowed directly on Int, but after boxing to Any it compares references:
    val boxA: Any = a
    val boxB: Any = b
    // On an ordinary JDK these are two distinct Integer objects (outside the -128..127 cache),
    // so the boxes are not identical. On a Valhalla JDK Integer is a value class with no identity,
    // so '===' compares by value and this assertion no longer holds.
    if (boxA === boxB) return "Fail: boxes are identical"
    return "OK"
}
```

### `primitivesAndFullValueClasses`

In addition to the primitives, this mode compiles **full value classes** — the multi-field value classes
introduced in
[KEEP-0454](https://github.com/Kotlin/KEEP/blob/main/proposals/KEEP-0454-better-immutability-value-classes-MFVC.md) —
as Valhalla value classes. The **inline** (`@JvmInline`) value classes are deliberately left as ordinary
reference/identity classes from the JVM's point of view.

In this mode, `Point` (a full value class) loses its identity — its identity marker is cleared and its fields
become strict — while the inline value class `Id` keeps its identity and its field stays non-strict:

```kotlin
value class Point(val x: Int, val y: Int)   // a Valhalla value class — no identity, strict fields
@JvmInline value class Id(val value: Int)    // still a reference class — keeps identity
```

The reason not to migrate inline value classes here is backward compatibility with their **boxes**. An inline
value class has a boxed reference form, and that box currently has JVM identity. Kotlin forbids relying on it —
you cannot use `===` (or `System.identityHashCode`, or synchronization) on an inline value class value
directly — but the identity is still *discoverable through workarounds*, namely by upcasting to `Any` first:

```kotlin
@JvmInline value class Id(val value: Int)

fun check(): String {
    val id = Id(1)
    // id === id                 // forbidden by the compiler: '===' on a value class
    val boxA: Any = id           // boxing produces a fresh reference box
    val boxB: Any = id           // a second boxing produces another, distinct box
    // Upcast to Any, the box identity becomes observable again:
    if (boxA === boxB) return "Fail: boxes coincided"
    // System.identityHashCode(boxA) likewise still exposes a per-box identity here.
    return "OK"
}
```

Keeping inline value classes as reference classes preserves this box identity — which is not meant to be relied
on, but remains observable — so code that depends on it through such workarounds keeps its current behavior.
Migrating them to Valhalla value classes, where boxes have no identity, is what the next mode does.

### `allValues`

The widest mode: it compiles the primitives, the full value classes, **and** the inline value classes as
Valhalla value classes. Now even the inline value class `Id` loses its identity, exactly like the full value
class `Point`.

This is the mode that should be used in practice to **try Project Valhalla out with Kotlin**: it migrates
everything that is semantically identity-free, so the whole program runs on value objects and can benefit from
flattening and scalarization.

Interoperability with **Java value classes** is also supported in this mode, in both directions.

A Kotlin `value class` is seen by Java as a value class:

```kotlin
// Kotlin
value class Point(val x: Int, val y: Int)
```

```java
// Java
Point p = new Point(1, 2);
boolean isValue = Point.class.isValue();   // true
```

And a Java `value class` / `value record` is usable from Kotlin as an ordinary type — the compiler recognizes
its value-ness (from the Java `value` modifier) and treats it accordingly, including listing it in
`LoadableDescriptors` when it appears as a field type:

```java
// Java
public value record Complex(double re, double im) {}
```

```kotlin
// Kotlin
fun magnitudeSquared(c: Complex): Double = c.re() * c.re() + c.im() * c.im()
```

Java value classes are also visible through Kotlin reflection: `KClass.isValue` reports `true` for them, and
their members remain invocable via `KFunction.call`/`callBy`.

# What the support includes

## Marking value classes and identity classes

The core of the feature is deciding, per mode, which declarations are Valhalla value classes and emitting the
corresponding class files:

- **Value classes** are emitted without the identity marker (`ACC_SUPER` cleared), which is how the JVM
  recognizes them (`Class.isValue()` becomes `true`), and their instance fields are marked `strict`.
- **Identity classes** — ordinary classes, and, in the `primitives`/`primitivesAndFullValueClasses` modes,
  inline value classes — keep `ACC_SUPER` and non-strict fields, i.e. exactly the representation Kotlin emits
  today. Marking a class as *not* a value class is as much a part of the support as marking one as a value
  class.
- **Abstract and sealed value class hierarchies** are supported: an abstract value class compiles to an
  abstract value class, and a sealed value class keeps its `permittedSubclasses`, so
  `Class.isSealed`/`permittedSubclasses` reflection continues to work.

## Interoperability with Java value classes

The support wires value-ness through the whole Java-interop surface so that Kotlin and Java value classes are
mutually usable:

- Java `value class`/`value record` declarations are recognized as value classes when read from source
  and from compiled class files.
- Kotlin value classes are exposed to Java as value classes (`Class.isValue()`), across module
  boundaries.
- Reflection agrees with the class file: `kotlin.reflect.KClass.isValue` classifies Java value classes through
  the descriptor and `java.lang.Class.isValue()`, and their members are callable through Kotlin reflection.

## The `LoadableDescriptors` attribute

For the JVM to preload and flatten a value-typed field, the class that declares the field must advertise the
field's type in the JEP 401 `LoadableDescriptors` attribute. The compiler emits this attribute — matching
javac's behavior — listing the descriptors of value-class and value-based field types. The attribute:

- is emitted on regular (identity) classes, on value classes, and on the file class that holds a static
  value-typed field;
- lists value-class fields, boxed primitives, and other JDK value-based types (`LocalDate`, `Optional`, …),
  but not identity reference types — `String`, or `Number` (an abstract class that is not itself value-based,
  even though its wrapper subtypes are) — nor bare primitives (`int`);
- when a value class has a field of its own type, that own type is left out of the list (the class is already
  being loaded), while the class's other value-typed fields are still listed; and when two value classes each
  hold a field of the other's type, each one lists the other. Both rules match javac.

## `LoadableDescriptors` for `@jdk.internal.ValueBased` and `@WillBecomeValue`

The JDK marks a number of stdlib types (`Integer`, `Optional`, `LocalDate`, …) with the internal
`@jdk.internal.ValueBased` annotation: they are value-like and are intended to become value classes, but in the
shipped stdlib they are still compiled as ordinary classes. The compiler lists such types in the
`LoadableDescriptors` attribute of the classes that have fields of those types, even though it does not itself
compile them as value classes. This choice is deliberate:

- **It matches javac.** javac emits the same attribute for the same value-based field types, so the bytecode
  Kotlin produces is consistent with the platform's.
- **It produces more efficient code.** Listing a value-based type in `LoadableDescriptors` is the preload hint
  the JVM needs to flatten and scalarize fields of that type once the type is a value class.
- **It works without recompiling the stdlib.** On a JDK that actually ships value-class versions of these
  stdlib types, the preload hint lets them be treated as value classes at load time — Kotlin does not need to
  recompile against, or re-mark, a value-class stdlib to get the benefit.

The same mechanism is the intended foundation for Kotlin's own analogue of `@jdk.internal.ValueBased`, the
proposed [`@WillBecomeValue`](https://github.com/Kotlin/KEEP/blob/main/proposals/KEEP-0468-will-become-value.md)
annotation: classes marked as *will become value* will be listed in the `LoadableDescriptors` attribute of
their users, so that they, too, gain the preload/flattening benefit ahead of an actual migration to
`value class` and without forcing a recompilation of their dependencies.
